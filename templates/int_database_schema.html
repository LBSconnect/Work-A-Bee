Welcome to the Work-A-Beez Database Design & Data Dictionary.

This document defines the logical database architecture, naming standards, entity relationships, data ownership, indexing strategy, auditing model, and data dictionary used throughout the Work-A-Beez platform.

It serves as the authoritative reference for database design, application development, reporting, integrations, and future schema evolution.

Database Philosophy

The Work-A-Beez database is designed around the following principles:

  • Data integrity
  • Scalability
  • Performance
  • Simplicity
  • Auditability
  • Tenant isolation
  • Extensibility

Every table should support long-term maintainability while minimizing duplication and enforcing business rules through appropriate constraints.

Design Principles

The database follows these architectural standards:

  • Normalize data where practical
  • Use surrogate primary keys
  • Enforce referential integrity
  • Store timestamps in UTC
  • Avoid duplicate data
  • Soft delete business records
  • Audit all write operations
  • Support horizontal scaling
  • Separate configuration from transactional data

Multi-Tenant Architecture

Every business entity belonging to a customer organization must include an Organization identifier.

Example:

Organization | +-- Employees +-- Departments +-- Locations +-- Schedules +-- Time Entries +-- Leave Requests +-- Reports +-- Notifications

All queries executed by the application must be scoped to the authenticated organization unless explicitly performing platform administration.

Naming Standards

Database object names use PascalCase for tables and camelCase or snake_case may be used consistently for columns depending on implementation standards adopted by the engineering team.

Examples:

Tables

Organization Employee Department Location Schedule Shift TimeEntry LeaveRequest Notification

Columns

employeeId organizationId departmentId createdAt updatedAt

Avoid abbreviations unless they are universally recognized.

Standard Columns

Every business table should include the following audit columns:

Column Purpose
id Primary Key
organizationId Tenant Identifier
createdAt Record creation timestamp
createdBy User that created record
updatedAt Last modification timestamp
updatedBy User that modified record
deletedAt Soft delete timestamp
deletedBy User performing deletion
version Optimistic concurrency version

Using a consistent audit model simplifies reporting, troubleshooting, and compliance.

Primary Keys

All tables use surrogate primary keys.

Example:

employeeId UUID

Characteristics:

  • Immutable
  • Unique
  • Non-business value
  • Never reused
  • Generated by the platform

Business identifiers such as employee numbers should not be used as primary keys.

Foreign Keys

Relationships are enforced using foreign keys where appropriate.

Example:

``` Employee -- departmentId

Department -- departmentId ```

Foreign keys should prevent orphaned records while supporting valid lifecycle operations.

Soft Delete Strategy

Business records are not permanently deleted under normal operations.

Instead, records are marked using:

deletedAt deletedBy

Soft deletion preserves:

  • Audit history
  • Payroll history
  • Reporting accuracy
  • Regulatory compliance
  • Historical analytics

Permanent deletion should be restricted to controlled administrative or legal processes.

Audit Model

Every create, update, and delete operation should generate an audit event.

Audit events capture:

  • User
  • Organization
  • Timestamp
  • Entity
  • Entity Identifier
  • Action
  • Previous Values
  • New Values
  • IP Address
  • Device Information (where available)

Audit records are append-only and must not be modified after creation.

Database Schema Overview

Core business tables include:

Organization User Role Permission Employee Department Location JobTitle ManagerAssignment Schedule Shift TimeEntry Break LeaveType LeaveBalance LeaveRequest Holiday Availability Announcement Message Notification PayrollPeriod AuditLog Document Attachment ApiKey Webhook Subscription Invoice

Additional lookup and configuration tables support platform operations.

Organization Table

Purpose:

Stores customer organization information.

Primary fields include:

  • organizationId
  • organizationName
  • legalName
  • website
  • timezone
  • language
  • subscriptionStatus
  • createdAt

Relationships:

  • One Organization has many Employees.
  • One Organization has many Departments.
  • One Organization has many Locations.
  • One Organization has many Schedules.

Employee Table

Purpose:

Stores employee records.

Primary fields include:

  • employeeId
  • organizationId
  • employeeNumber
  • firstName
  • lastName
  • preferredName
  • email
  • phone
  • hireDate
  • employmentStatus
  • managerId
  • departmentId
  • locationId
  • jobTitleId

Relationships:

Employee belongs to:

  • Organization
  • Department
  • Location
  • Job Title

Employee has:

  • Time Entries
  • Leave Requests
  • Availability
  • Notifications
  • Documents
  • Schedule Assignments

Department Table

Purpose:

Groups employees for operational management.

Key fields:

  • departmentId
  • organizationId
  • departmentName
  • managerId
  • description

Relationships:

Department has many Employees.

Department has many Schedules.

Department has many Reports.

Location Table

Purpose:

Represents physical or virtual work locations.

Fields include:

  • locationId
  • organizationId
  • locationName
  • address
  • city
  • state
  • postalCode
  • country
  • timezone
  • latitude
  • longitude

Relationships:

Location contains:

  • Employees
  • Shifts
  • Time Entries
  • Schedules

Schedule Table

Purpose:

Represents a published schedule.

Fields include:

  • scheduleId
  • organizationId
  • name
  • startDate
  • endDate
  • status
  • publishedAt
  • createdBy

Relationships:

Schedule contains many Shifts.

Shift Table

Purpose:

Represents a scheduled work assignment.

Fields include:

  • shiftId
  • scheduleId
  • employeeId
  • locationId
  • departmentId
  • startTime
  • endTime
  • status
  • notes

Relationships:

Shift belongs to one Schedule.

Shift belongs to one Employee.

Shift belongs to one Department.

TimeEntry Table

Purpose:

Stores actual worked time.

Fields include:

  • timeEntryId
  • employeeId
  • clockIn
  • clockOut
  • totalHours
  • overtimeHours
  • approvalStatus

Relationships:

TimeEntry belongs to Employee.

TimeEntry belongs to Payroll Period.

Relationship Standards

Relationships should use consistent naming conventions.

Examples:

Organization | +-- Employee | +-- TimeEntry | +-- LeaveRequest | +-- Availability | +-- Notification | +-- Department | +-- Location | +-- Schedule +-- Shift

Relationship names should clearly communicate ownership and cardinality.

Indexing Standards

Indexes should be created for:

  • Primary Keys
  • Foreign Keys
  • Frequently searched columns
  • Frequently filtered columns
  • Frequently joined columns
  • Unique business identifiers

Composite indexes should reflect common query patterns identified through monitoring.

Constraints

Appropriate database constraints should be used to enforce data integrity.

Examples include:

  • Primary Key Constraints
  • Foreign Key Constraints
  • Unique Constraints
  • Check Constraints
  • Not Null Constraints

Business validation should primarily occur in the application layer, with the database providing an additional layer of protection.

Transaction Management

Database transactions should be:

  • Atomic
  • Consistent
  • Isolated
  • Durable (ACID)

Long-running transactions should be avoided to reduce contention and improve scalability.

Performance Guidelines

To maintain performance as the platform grows:

  • Retrieve only required columns.
  • Use pagination for large result sets.
  • Avoid N+1 query patterns.
  • Monitor slow queries.
  • Optimize indexes based on real-world usage.
  • Archive historical data when appropriate.

Regular performance reviews should be part of ongoing database maintenance.

Data Retention

Retention policies should align with legal, regulatory, and contractual requirements.

Examples:

  • Audit Logs
  • Payroll Records
  • Time Entries
  • Leave Requests
  • Employee Documents

Retention periods may vary by jurisdiction and customer configuration.

Change Management

All schema changes should follow the organization's change management process.

Recommended practices include:

  • Version-controlled migration scripts
  • Peer review
  • Automated testing
  • Rollback procedures
  • Staging validation
  • Production deployment approvals

Database changes should be backward compatible whenever practical.
