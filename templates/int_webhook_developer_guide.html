Welcome to the Work-A-Beez Webhook Developer Guide.

Work-A-Beez webhooks enable applications to receive near real-time notifications when events occur within the platform. Rather than polling the API for changes, subscribed applications receive HTTPS POST requests whenever selected events occur.

This guide explains how to subscribe to webhook events, validate requests, process deliveries, handle retries, and build reliable event-driven integrations.

Introduction

A webhook is an HTTP callback triggered automatically when a specified event occurs.

Instead of requesting updates repeatedly, your application receives notifications as events happen.

Typical webhook use cases include:

  • Employee onboarding
  • Schedule publication
  • Shift assignment
  • Clock in/out activity
  • Leave approvals
  • User invitations
  • Payroll exports

How Webhooks Work

The webhook lifecycle follows these steps:

  1. Register a webhook endpoint.
  2. Subscribe to one or more event types.
  3. Work-A-Beez detects an event.
  4. A signed HTTPS POST request is sent.
  5. Your application validates the request.
  6. Your application processes the payload.
  7. Your application returns a successful HTTP response.

If delivery fails, Work-A-Beez retries according to the configured retry policy.

Architecture

Webhook communication is asynchronous.

Work-A-Beez │ │ Event Occurs ▼ Webhook Dispatcher │ HTTPS POST ▼ Customer Endpoint │ Validate Signature │ Queue Processing │ Business Logic

Endpoints should acknowledge receipt quickly and defer lengthy processing to background workers.

Creating Webhook Subscriptions

Administrators can create webhook subscriptions from:

Administration → Developer Settings → Webhooks

Each subscription requires:

  • Endpoint URL
  • Event selection
  • Authentication configuration
  • Secret key
  • Description (optional)
  • Status (Enabled/Disabled)

Only HTTPS endpoints are supported.

Event Payload Structure

All webhook payloads follow a consistent format.

Example:

json { "eventId": "evt_8f2d91c4", "eventType": "employee.created", "occurredAt": "2026-07-23T14:30:18Z", "organizationId": "org_10293", "resourceId": "emp_45082", "data": { "employeeId": "emp_45082", "firstName": "John", "lastName": "Smith", "department": "Operations" } }

Applications should ignore unknown properties to maintain forward compatibility.

Authentication & Signature Verification

Each webhook request includes a digital signature to verify authenticity.

Recommended validation process:

  1. Read the raw request body.
  2. Retrieve the webhook secret.
  3. Generate the expected signature.
  4. Compare it with the signature provided in the request header.
  5. Reject requests with invalid signatures.

Never process unsigned or unverifiable requests.

Required Headers

Typical webhook headers include:

http Content-Type: application/json X-WorkABeez-Event X-WorkABeez-Signature X-WorkABeez-Timestamp X-WorkABeez-DeliveryId

Applications should validate required headers before processing.

Successful Responses

After successfully receiving and validating a webhook, return:

http HTTP/1.1 200 OK

or

http HTTP/1.1 204 No Content

Responses should be returned promptly to avoid unnecessary retries.

Retry Policy

If a delivery attempt fails, Work-A-Beez retries automatically.

Typical retry schedule:

Attempt Delay
1 Immediate
2 1 minute
3 5 minutes
4 15 minutes
5 1 hour
6 6 hours

Retries may stop after the maximum configured attempts.

Idempotency

Webhook consumers should process events idempotently.

Use the eventId to detect duplicate deliveries.

Example logic:

If eventId already processed Ignore request Else Process event

Duplicate deliveries are expected behavior during retry scenarios.

Event Ordering

Webhook delivery order is not guaranteed.

Applications should:

  • Handle out-of-order events gracefully.
  • Retrieve current resource state from the API if sequencing is critical.
  • Avoid assuming that related events arrive consecutively.

Security Best Practices

To secure webhook endpoints:

  • Require HTTPS.
  • Verify request signatures.
  • Restrict accepted HTTP methods to POST.
  • Validate timestamps to reduce replay attacks.
  • Rotate webhook secrets periodically.
  • Log delivery identifiers for troubleshooting.
  • Rate-limit incoming requests where appropriate.

Do not expose webhook endpoints unnecessarily.

Testing Webhooks

Before enabling production subscriptions:

  • Verify endpoint availability.
  • Validate signatures.
  • Test retry behavior.
  • Simulate malformed payloads.
  • Confirm duplicate event handling.
  • Verify timeout handling.

Testing should include both successful and failure scenarios.

Troubleshooting

Signature Validation Failed

Verify:

  • Shared secret
  • Raw request body
  • Hash algorithm
  • Header values

Duplicate Events

Expected during retries.

Use the event identifier to prevent duplicate processing.

Missing Events

Check:

  • Subscription status
  • Event selection
  • Endpoint availability
  • Delivery logs
  • Firewall rules

Timeout Errors

Ensure:

  • Endpoint responds within expected time.
  • Long-running work is processed asynchronously.
  • Background workers handle intensive tasks.
