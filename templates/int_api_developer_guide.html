Welcome to the Work-A-Beez API Developer Guide.

The Work-A-Beez API allows developers to securely integrate workforce management functionality with third-party applications, payroll providers, HR systems, accounting platforms, identity providers, and custom business applications.

Whether you're building a custom dashboard, synchronizing employee records, automating workflows, or integrating payroll, the API provides secure and scalable access to Work-A-Beez resources.

Introduction

The Work-A-Beez API is built using REST principles and communicates using HTTPS.

Features include:

  • RESTful endpoints
  • JSON request and response bodies
  • Secure authentication
  • OAuth 2.0 support
  • Webhooks
  • Role-based authorization
  • Pagination
  • Filtering
  • Versioning
  • Audit logging

The API is designed to remain consistent across all resources to simplify development and maintenance.

API Overview

The API enables secure access to organizational data and operations based on the permissions granted to the authenticated user or application.

Common integration scenarios include:

  • Employee synchronization
  • Payroll exports
  • Timecard retrieval
  • Schedule management
  • Leave requests
  • Attendance reporting
  • Identity management
  • Business intelligence dashboards
  • Mobile applications
  • Custom workflow automation

Base URLs

Production

https://api.workabeez.net/v1

Sandbox

https://sandbox-api.workabeez.net/v1

All API communication must occur over HTTPS.

Requests made using HTTP are rejected.

Authentication

Every API request must be authenticated.

Supported authentication methods include:

  • API Keys
  • OAuth 2.0
  • Bearer Tokens

Unauthenticated requests receive an HTTP 401 Unauthorized response.

API Keys

API keys allow server-to-server communication for trusted applications.

Administrators may generate API keys by navigating to Administration → Developer Settings → API Keys.

Each API key is associated with:

  • Organization
  • Environment
  • Creation Date
  • Expiration Date (optional)
  • Assigned Scopes
  • Status

API keys should be treated as confidential credentials.

Never expose API keys in:

  • Client-side JavaScript
  • Mobile applications
  • Public repositories
  • Documentation examples containing live credentials

OAuth 2.0

Work-A-Beez supports OAuth 2.0 for applications acting on behalf of users.

Supported grant types include:

  • Authorization Code
  • Client Credentials
  • Refresh Token

Organizations may restrict OAuth applications based on internal security policies.

Authorization

Authentication identifies the caller.

Authorization determines what the caller may access.

Every request is evaluated against:

  • Organization
  • User Role
  • Assigned Permissions
  • Resource Ownership
  • API Scopes

Access to unauthorized resources returns:

403 Forbidden

API Scopes

Examples include:

  • Employee.Read
  • Employee.Write
  • Schedule.Read
  • Schedule.Write
  • Timecard.Read
  • Timecard.Write
  • Leave.Read
  • Leave.Write
  • Reports.Read
  • Reports.Export
  • Admin.FullAccess

Applications should request only the scopes required for their intended functionality.

Content Types

Requests must specify:

Content-Type: application/json

Responses are returned as:

application/json

File download endpoints may return other content types when appropriate.

Making Your First Request

Example request:

http GET /employees Authorization: Bearer YOUR_ACCESS_TOKEN

Successful response:

json { "data": [], "page": 1, "pageSize": 25, "total": 0 }

Request Structure

Typical requests contain:

  • HTTP Method
  • Endpoint
  • Headers
  • Query Parameters
  • Request Body

Example:

http POST /employees

json { "firstName": "John", "lastName": "Smith", "email": "john.smith@example.com" }

HTTP Methods

The API supports standard HTTP methods.

Method Purpose
GET Retrieve data
POST Create new resources
PUT Replace resources
PATCH Partially update resources
DELETE Remove resources

Method usage is consistent across all endpoints.

Response Format

Successful responses generally follow a consistent structure.

Example:

json { "success": true, "data": {}, "meta": {}, "links": {} }

Error responses return descriptive information to help developers diagnose issues.

Status Codes

Common HTTP status codes include:

Code Meaning
200 OK
201 Created
202 Accepted
204 No Content
400 Bad Request
401 Unauthorized
403 Forbidden
404 Not Found
409 Conflict
422 Validation Error
429 Too Many Requests
500 Internal Server Error
503 Service Unavailable

Applications should handle each status code appropriately.

Error Responses

Error responses include a standardized structure.

Example:

json { "success": false, "error": { "code": "ValidationError", "message": "Email address is required.", "details": [ { "field": "email", "message": "This field is required." } ] } }

The code field is intended for programmatic handling, while the message field is human-readable.

Pagination

Endpoints returning collections support pagination.

Common query parameters include:

?page=1&pageSize=25

Responses include metadata such as:

  • Current Page
  • Page Size
  • Total Records
  • Total Pages
  • Next Page
  • Previous Page

Pagination helps improve performance and reduces response size.

Filtering

Many endpoints support filtering using query parameters.

Examples include:

?department=Sales

?status=Active

?location=Houston

Multiple filters may be combined to narrow results.

Sorting

Results may be sorted in ascending or descending order.

Examples:

?sort=lastName

?sort=-hireDate

A leading minus sign (-) indicates descending order.

Rate Limits

To maintain platform stability, API requests are subject to rate limits.

When a limit is exceeded, the API returns:

429 Too Many Requests

Responses may include rate limit headers indicating:

  • Request Limit
  • Remaining Requests
  • Reset Time

Applications should implement retry logic with exponential backoff where appropriate.

Idempotency

For operations that create or modify resources, clients may provide an idempotency key to prevent accidental duplicate processing during retries.

Example header:

Idempotency-Key: 6b7f2c1e-1234-5678-90ab-cdef12345678

The server uses this value to safely identify repeated requests.

API Versioning

The Work-A-Beez API uses URI-based versioning.

Example:

https://api.workabeez.net/v1

New API versions are introduced only for changes that are not backward compatible.

Whenever practical, existing versions remain available for a transition period to support customer migrations.
